# SwiftHelpCenter 定位与路线图

SwiftHelpCenter 是一个面向 Swift App 开发者的开源帮助中心套件。

它对外仍然叫“帮助中心”，因为这个名字最符合用户和开发者的心智：用户知道可以在这里寻求帮助，开发者也知道它是可以集成到 App 内的帮助入口。

但从产品定位上看，SwiftHelpCenter 不只是一个传统帮助页面，也不是 App 设置功能的替代品。它更接近一套可复用的 **App 用户沟通基础设施**：帮助开发者用更低成本，在 App 内和用户建立持续沟通。

## 核心定位

SwiftHelpCenter 的目标是让开发者快速为自己的 App 增加一套完整、可维护、可扩展的帮助中心。

它重点解决这些问题：

- 用户遇到问题时，能找到技术支持入口。
- 用户有建议或问题时，能方便地反馈给开发者。
- App 更新后，开发者能向用户说明更新内容、视频教程和相关文章。
- 用户认可 App 时，能自然地去评分或留下评价。
- 开发者可以持续维护帮助、反馈、公告、更新说明等沟通内容，而不必每个 App 都重复造轮子。

## 不是什么

SwiftHelpCenter 不打算替代传统 App 设置页面。

设置页面通常用于管理 App 配置，例如账号、偏好、权限、主题、存储、快捷键等。帮助中心则聚焦于用户沟通、支持、反馈和版本说明。

两者可以相互连接，但职责应该保持清晰：

- 设置页面：管理 App 行为。
- 帮助中心：建立用户沟通。

## 当前能力

目前 SwiftHelpCenter 已经包含以下基础能力：

- 帮助中心入口
- 版本历史
- 更新说明
- 视频链接
- 快速入口
- FAQ
- 技术支持网站入口
- 用户反馈页面
- App 评分入口
- 使用一段时间后的评分提醒机制
- macOS / iOS 双平台支持
- 内置 Design System

这些能力构成了帮助中心的第一阶段：让开发者可以在 App 内快速建立基础用户沟通入口。

## 设计原则

后续功能扩展应遵循这些原则：

1. **保持帮助中心心智**

   对用户来说，它仍然是帮助中心；对开发者来说，它仍然是一个容易集成的 Swift Package。

2. **围绕沟通场景扩展**

   新功能应优先服务于支持、反馈、公告、更新说明、教程、路线图、用户调查等沟通场景。

3. **不变成设置页**

   不把通用 App 设置功能塞进帮助中心。帮助中心可以链接到设置页，但不替代设置页。

4. **开发者负担要低**

   配置方式应尽量简单，默认体验应足够可用。复杂能力可以逐步开放，但不能让基础使用变复杂。

5. **macOS 和 iOS 都要可用**

   包应持续支持 macOS 和 iOS，界面可以根据平台差异做不同排版。

6. **开源项目要克制承诺**

   Roadmap 用于表达方向，不代表所有远期功能都会实现。

## 近期路线图

近期功能是大概率会实现的方向，目标是补齐帮助中心作为“用户沟通入口”的核心体验。

### 1. 公告功能

让开发者可以在帮助中心展示重要通知。

典型场景：

- 服务维护通知
- 临时问题说明
- 新功能提醒
- 活动或优惠信息
- 重要兼容性提醒

可能的数据结构：

- 标题
- 内容
- 发布时间
- 优先级
- 是否置顶
- 可选链接
- 是否已读

### 2. What’s New 更新弹窗

App 更新后，第一次打开时展示本次最重要的变化。

它和版本历史不同：版本历史适合完整记录，What’s New 更适合突出 1 到 3 个用户最该知道的变化。

典型场景：

- 新版本发布后弹出更新说明
- 展示新功能入口
- 引导用户查看视频教程
- 提醒用户某个工作流已经变化

### 3. 帮助内容分类

当前帮助中心内容较少时，用列表即可。后续内容增多后，需要支持更清晰的分类。

可能分类：

- 版本历史
- 常见问题
- 视频教程
- 技术支持
- 反馈与建议
- 公告

### 4. 更完善的反馈体验

反馈页面可以继续增强：

- 反馈类型选择
- 用户邮箱或联系方式
- 附件上传
- 反馈成功后的确认状态
- 不同平台的反馈渠道配置

目标不是做一个完整客服系统，而是让独立开发者能更容易收到有效反馈。

### 5. 远程内容配置的基础能力

帮助中心里的某些内容如果每次都要发版才能更新，成本会偏高。

近期可以先探索轻量远程配置：

- 从远程 JSON 加载版本历史
- 从远程 JSON 加载 FAQ
- 从远程 JSON 加载公告
- 本地配置作为 fallback

这会让帮助中心从“静态组件”逐步走向“可运营的沟通入口”。

## 远期方向

远期方向代表可能值得探索的增强能力，但不保证都会实现。

### 1. 用户调查

让开发者可以在帮助中心中发起轻量问卷。

典型问题：

- 你最想要哪个功能？
- 你使用这个 App 的主要场景是什么？
- 这个功能是否解决了你的问题？

### 2. 功能投票

让用户对候选功能投票，帮助开发者判断需求优先级。

可能能力：

- 候选功能列表
- 投票按钮
- 简单排序
- “我也需要”计数

### 3. 公开 Roadmap

让开发者展示计划中的功能、正在开发的功能和已经完成的功能。

这可以增强用户信任，也能让反馈形成闭环。

### 4. 反馈状态跟踪

未来可以考虑让用户看到反馈状态：

- 已收到
- 已确认
- 已计划
- 已修复
- 已发布

这个方向价值很高，但实现复杂度也更高，需要谨慎评估。

### 5. Changelog 与反馈关联

将版本更新和用户反馈关联起来。

例如某个版本说明中可以标注：

- 修复了某个用户反馈的问题
- 采纳了某个用户建议
- 增加了用户投票最多的功能

### 6. 用户分群展示

不同用户可能需要不同帮助内容。

例如：

- 新用户看到入门教程
- 老用户看到高级功能
- 专业用户看到 API 或自动化说明
- 免费用户和付费用户看到不同支持入口

这个能力可能很有用，但也容易增加配置复杂度，应作为远期探索。

### 7. 内容运营能力

帮助中心未来可以成为 App 内的轻量内容运营入口。

可能包括：

- 教程文章
- 视频合集
- 使用案例
- 最佳实践
- 多平台内容链接

## 分阶段目标

### 第一阶段：可复用帮助中心

目标：让开发者能快速集成一个体面的帮助中心。

包括：

- 版本历史
- FAQ
- 视频链接
- 技术支持
- 反馈页面
- App 评分入口

### 第二阶段：主动沟通中心

目标：让开发者不只是被动接收反馈，也能主动告诉用户重要信息。

包括：

- 公告
- What’s New
- 远程内容配置
- 更好的帮助内容分类

### 第三阶段：反馈闭环

目标：让用户感觉自己的反馈被看到，也让开发者更容易判断优先级。

可能包括：

- 用户调查
- 功能投票
- 公开 Roadmap
- 反馈状态
- 更新说明与反馈关联

## 对外表达建议

对用户：

> 帮助中心可以查看更新说明、教程视频、技术支持和反馈入口。

对开发者：

> SwiftHelpCenter 是一个 Swift Package，帮助你快速为 macOS / iOS App 集成帮助中心、版本历史、反馈、评分提醒和用户沟通入口。

对项目长期定位：

> SwiftHelpCenter 希望成为独立开发者和小团队可复用的 App 内用户沟通套件。

## 当前优先级建议

接下来最值得优先实现的是：

1. 公告功能
2. What’s New 更新弹窗
3. 帮助内容分类
4. 远程 JSON 配置

这四项和当前能力连接最自然，开发成本相对可控，也最能强化“帮助中心是用户沟通入口”的产品定位。
